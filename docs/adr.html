ADR-001 Use Bun as the JavaScript runtime
M1 Accepted
Context

We need a runtime to execute TypeScript code for a CLI coding agent. The runtime must support TypeScript natively, start up fast (it's a CLI tool — startup latency is visible to the user), and have modern built-in APIs so we don't need dozens of extra packages.

Decision
Use Bun as the runtime, package manager, and bundler for the entire project.
Alternatives considered
Option TypeScript native Startup speed Built-in APIs Ecosystem
Bun ✓ ✓ Yes ✓ ~7ms ✓ file, fetch, spawn, SQLite ✓ npm compatible
Node.js ✗ Needs ts-node / tsx ~ ~100ms ~ fs, http (verbose) ✓ Largest ecosystem
Deno ✓ Yes ✓ Fast ✓ Built-in fetch, file ✗ Different module system, smaller ecosystem
Consequences
+No build step — bun run file.ts just works
+Single tool replaces Node + npm + ts-node + esbuild
+Fast startup improves CLI feel significantly
+Bun.file(), Bun.spawn(), Bun.$ cover all agent needs natively
−A small number of Node.js packages may not be fully compatible
−Bun is newer — less StackOverflow coverage than Node
ADR-002 Use Ink for the terminal UI
M1 Accepted
Context

The agent needs a rich interactive terminal UI — scrollable message history, a live input box with cursor, status bar, streaming text, syntax-highlighted code blocks, and diff views. We need a framework that makes this manageable without low-level terminal escape code manipulation.

Decision
Use Ink (React for the terminal) as the UI framework.
Alternatives considered
Option Abstraction level Component model Hooks / state
Ink ✓ ✓ High — React components ✓ Full React tree ✓ useState, useReducer, useEffect
blessed / neo-blessed ~ Mid — widget API ✗ Imperative, no components ✗ Manual event wiring
Raw ANSI escape codes ✗ None — fully manual ✗ None ✗ None
Consequences
+React knowledge transfers — same hooks, same mental model as web React
+Declarative UI means no manual screen redraw logic
+Flexbox layout via Box — no manual column arithmetic
+Used by real CLIs including Vercel, Gatsby, and Claude Code itself
−Requires learning React if you don't know it already
−Ink re-renders the whole screen on state change — must profile for large outputs
ADR-003 Use TypeScript in strict mode
M1 Accepted
Context

A coding agent handles file operations, shell commands, and API calls — categories where type errors cause real damage (wrong path, wrong parameter, mishandled null). We need the type system to catch as many bugs as possible at compile time rather than at runtime.

Decision
Enable "strict": true in tsconfig.json, plus noUnusedLocals and noUnusedParameters.
What strict mode enables
Included flags
✓strictNullChecks — null and undefined are not assignable to other types
✓noImplicitAny — variables must have explicit or inferred types
✓strictFunctionTypes — function parameter types checked correctly
✓strictPropertyInitialization — class fields must be initialized
Extra flags added
✓noUnusedLocals — error on declared but unused variables
✓noUnusedParameters — error on declared but unused function parameters
Consequences
+Null-related crashes caught at compile time — critical for a file/shell agent
+Dead code detected automatically — keeps the codebase clean
−More verbose in places — requires explicit null handling and type annotations
−Higher barrier for quick prototyping — intentional trade-off
ADR-004 Use Biome for linting and formatting
M1 Accepted
Context

The project needs consistent code formatting and a linter to catch common mistakes. The traditional TypeScript toolchain uses ESLint (linting) + Prettier (formatting) as two separate tools with two separate configs that sometimes conflict with each other.

Decision
Use Biome — a single tool that replaces both ESLint and Prettier. Run via bun biome check --write.
Alternatives considered
Option Tools needed Config files Speed
Biome ✓ ✓ 1 ✓ 1 (biome.json) ✓ ~35× faster than ESLint
ESLint + Prettier ✗ 2 + many plugins ✗ .eslintrc + .prettierrc ~ Slower, especially on large files
Consequences
+One config file, one command, zero conflicts between formatter and linter
+Written in Rust — dramatically faster CI times
−Biome is newer — a handful of ESLint rules don't have Biome equivalents yet
ADR-005 Use Zod for runtime validation and tool schemas
M3 Proposed
Context

The agentic loop receives tool call inputs from the LLM as raw JSON. These must be validated before execution — a wrong type or missing field on a file-write tool could cause real damage. We also need to generate JSON Schema definitions from the same type definitions, so the LLM knows how to call each tool correctly.

Decision
Use Zod to define tool input schemas. Use z.infer<typeof Schema> to derive TypeScript types. Auto-generate JSON Schema for the Anthropic API from the same Zod schema.
Python parallel

Zod is the TypeScript equivalent of Pydantic. z.object({}) ≈ BaseModel, schema.parse(data) ≈ model.model_validate(data), schema.safeParse(data) ≈ catching ValidationError. The key advantage over Pydantic: z.infer derives the TypeScript type automatically — one source of truth for both the type and the validator.

Consequences
+Single Zod schema generates both the TypeScript type AND the JSON Schema for the LLM
+Runtime validation catches malformed LLM tool calls before they execute
+Familiar API for anyone coming from Pydantic
−Adds a dependency and a new API to learn
−Zod v4 (2025) has breaking changes from v3 — pin the version
ADR-006 Use discriminated unions for all state actions
M1 Accepted
Context

The application has several state machines — the input buffer, vim mode, agent loop status, and streaming state. Each needs a set of operations that update state in well-defined ways. We need a pattern that is type-safe, exhaustively checked by the compiler, and readable to future contributors.

Decision
Model all state updates as discriminated union action types handled by a reducer function, consumed via React's useReducer hook.
Pattern example
Without pattern
−setState scattered across many handlers
−No central place to understand all state transitions
−Easy to forget a case, no compile-time exhaustiveness check
With pattern
+All transitions in one reducer function
+TypeScript enforces exhaustive switch — missing case = compile error
+Actions are named and self-documenting: { type: "deleteWordBackward" }
Consequences
+Pure reducer functions are trivially testable — no React needed in tests
+State never mutated directly — predictable, debuggable, undo-able
+Pattern scales to the agent loop state machine in Milestone 3
−More boilerplate than direct setState for simple cases
